6.0Warum eine Datenbank?

Eine Tabelle reicht für eine kleine Namensliste. In einer Schulbibliothek hängen jedoch verschiedene Dinge zusammen: Eine Person kann mehrere Bücher ausleihen, ein Buch kann im Laufe der Zeit von verschiedenen Personen ausgeliehen werden, und jede Ausleihe besitzt ein eigenes Datum. Würde alles in einer einzigen Tabelle stehen, müssten Namen und Buchtitel ständig wiederholt werden.

Tabellenkalkulation

Gut für Berechnungen, einzelne Listen und schnelle Diagramme.

Grenze: Zusammenhängende Daten werden leicht doppelt oder widersprüchlich.

Relationale Datenbank

Gut für viele verknüpfte Datensätze, eindeutige Regeln und wiederkehrende Auswertungen.

Stärke: Beziehungen werden über Schlüssel zuverlässig abgebildet.

Merke: Datenbank

Eine Datenbank ist eine strukturierte Sammlung zusammengehöriger Daten. Ein Datenbankmanagementsystem wie MySQL speichert diese Daten, prüft Regeln und beantwortet Abfragen.

6.1Die reale Welt als ER-Modell

Bevor Tabellen angelegt werden, wird die Situation vereinfacht. Das Entity-Relationship-Modell, kurz ER-Modell, zeigt die wichtigen Objekte, ihre Eigenschaften und ihre Beziehungen. Es ist der Bauplan der Datenbank.

6.2Vom ER-Modell zum Relationenmodell

Im Relationenmodell wird jeder Entitätstyp zu einer Relation, die später als Tabelle umgesetzt wird. Die n:m-Beziehung erhält eine eigene Relation AUSLEIHE. So wird aus dem Bild ein präziser Tabellenplan.

PERSON

Primärschlüssel (PK)

Identifiziert jeden Datensatz einer Tabelle eindeutig. Zwei Personen dürfen denselben Namen, aber niemals dieselbe person_id besitzen.

Fremdschlüssel (FK)

Verweist auf den Primärschlüssel einer anderen Tabelle. AUSLEIHE.person_id zeigt auf PERSON.person_id.

6.2+Redundanz und 3. Normalform

Redundanz bedeutet, dass dieselbe Information unnötig mehrfach gespeichert wird. In einer großen Ausleihtabelle würde Mias Name bei jeder Ausleihe erneut stehen. Ändert sich der Name, müssten viele Zeilen angepasst werden. Wird eine Zeile vergessen, widerspricht die Datenbank sich selbst.

ausleihe_idperson_nameklassebuch_titel
501Mia SommerJ1/1Python verstehen
502Mia SommerJ1/1Datenbanken kompakt
503Mia SommerJ1/1Algorithmen entdecken
1. NormalformJede Zelle enthält genau einen Wert, keine Listen oder Wiederholungsgruppen.
2. NormalformJedes Nichtschlüsselattribut hängt vom gesamten Primärschlüssel ab.
3. NormalformNichtschlüsselattribute hängen nur vom Primärschlüssel ab, nicht voneinander.
Typische Anomalien
  • Änderungsanomalie: Eine Information muss an vielen Stellen geändert werden.
  • Einfügeanomalie: Ein Buch kann erst gespeichert werden, wenn bereits eine Ausleihe existiert.
  • Löschanomalie: Mit der letzten Ausleihe verschwinden versehentlich auch die Buchdaten.
Prüffrage für die 3. Normalform

Beschreibt jedes Attribut wirklich das Objekt dieser Tabelle? Wenn ein Attribut eigentlich zu einem anderen Objekt gehört, braucht es meist eine eigene Tabelle und einen Fremdschlüssel.

6.3Die Datenbank mit SQL aufbauen

In MySQL Workbench wird das Modell mit SQL umgesetzt. SQL steht für Structured Query Language. Ein neues Skript wird geöffnet, ausgeführt und das Ergebnis anschließend im Schema-Bereich kontrolliert.

Tabellen erstellen

CREATE TABLE person (
  person_id INT PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  klasse VARCHAR(20) NOT NULL
);

CREATE TABLE buch (
  buch_id INT PRIMARY KEY,
  titel VARCHAR(150) NOT NULL,
  autor VARCHAR(100) NOT NULL
);

CREATE TABLE ausleihe (
  ausleihe_id INT PRIMARY KEY,
  person_id INT NOT NULL,
  buch_id INT NOT NULL,
  ausleihdatum DATE NOT NULL,
  rueckgabedatum DATE,
  FOREIGN KEY (person_id) REFERENCES person(person_id),
  FOREIGN KEY (buch_id) REFERENCES buch(buch_id)
);

Datensätze einfügen

INSERT INTO person (person_id, name, klasse)
VALUES (1, 'Mia Sommer', 'J1/1');

INSERT INTO buch (buch_id, titel, autor)
VALUES (10, 'Python verstehen', 'Lena Code');

INSERT INTO ausleihe
  (ausleihe_id, person_id, buch_id, ausleihdatum)
VALUES
  (501, 1, 10, '2026-09-14');
Datenkonsistenz

Die gespeicherten Daten erfüllen die vereinbarten Regeln und widersprechen sich nicht.

Referentielle Integrität

Ein Fremdschlüssel darf nur auf einen tatsächlich vorhandenen Datensatz zeigen. Eine Ausleihe für person_id = 99 wird abgelehnt, wenn Person 99 nicht existiert.

Daten aus mehreren Tabellen verbinden

SELECT person.name, buch.titel, ausleihe.ausleihdatum
FROM ausleihe
INNER JOIN person
  ON ausleihe.person_id = person.person_id
INNER JOIN buch
  ON ausleihe.buch_id = buch.buch_id
WHERE ausleihe.rueckgabedatum IS NULL
ORDER BY person.name;
Merke: JOIN

Ein JOIN verbindet passende Datensätze über Primär- und Fremdschlüssel. Ohne die ON-Bedingung würden alle Personen mit allen Büchern kombiniert. Das Ergebnis wäre fachlich falsch und sehr groß.

Gruppieren und rechnen

SELECT person.klasse,
       COUNT(*) AS anzahl_ausleihen
FROM ausleihe
INNER JOIN person
  ON ausleihe.person_id = person.person_id
WHERE YEAR(ausleihe.ausleihdatum) = 2026
GROUP BY person.klasse
ORDER BY anzahl_ausleihen DESC;

6.5Massendaten: erkennen, aber verantwortlich handeln

Viele einzelne Datensätze können Muster sichtbar machen. Bewegungsdaten zeigen beispielsweise Verkehrsströme, Besucherzahlen oder gefährliche Schulwege. Dieselben Daten können jedoch auch Personen überwachen oder falsche Schlüsse nahelegen.

ChanceZusammenhänge erkennen, Angebote verbessern, Ressourcen sinnvoll planen.
PrüfenIst die Datenbasis vollständig? Ist ein Zusammenhang wirklich eine Ursache?
RisikoÜberwachung, Diskriminierung, Datenmissbrauch und Entscheidungen ohne Kontext.
Korrelation ist nicht automatisch Ursache

Zwei Merkmale können gemeinsam auftreten, ohne dass eines das andere verursacht. Eine Datenbank liefert Muster. Die fachliche Erklärung und eine faire Bewertung brauchen zusätzlich Kontext.

Fünf Fragen für eine begründete Bewertung

  1. Zweck: Welches konkrete Problem soll gelöst werden?
  2. Datensparsamkeit: Welche Daten sind dafür wirklich notwendig?
  3. Transparenz: Wissen Betroffene, was gespeichert und ausgewertet wird?
  4. Qualität: Sind Daten vollständig, aktuell und frei von systematischen Verzerrungen?
  5. Folgen: Wer profitiert, wer trägt ein Risiko und wer kann widersprechen?

Übung 2 · Kardinalitäten lesenEinstieg

Bestimmen und begründen Sie die Kardinalität:

  1. KLASSE - SCHÜLER
  2. LEHRKRAFT - KURS
  3. KUNDIN - KUNDENKARTE
Lösung anzeigen
  1. 1:n, eine Klasse umfasst viele Schülerinnen und Schüler, jede Schülerin bzw. jeder Schüler gehört zu genau einer Klasse.
  2. n:m, wenn Team-Teaching möglich ist: Eine Lehrkraft unterrichtet mehrere Kurse und ein Kurs kann mehrere Lehrkräfte haben. Nimmt man dagegen an, dass jeder Kurs genau eine Lehrkraft hat, ist 1:n richtig – bei Kardinalitäten zählt immer die genannte Annahme!
  3. Im vereinfachten Modell 1:1: Eine Kundin besitzt genau eine Kundenkarte, jede Karte gehört genau einer Kundin.
